# Engineering Journal
## Date: 2026-01-01
## Author: Student Engineer (3rd Year ECE)

### Objective
Document the motivation and origin of the hardware acceleration project.

### Current Understanding
Recently, I studied the MeghDut project's telemetry architecture, particularly how drones communicate via ESP32 bridges using MAVLink over UDP or JSON over MQTT. The `HardwareGateway.js` node server currently handles all data normalization (lat, lng, alt, heading, battery, speed) and DigitalTwin tracking. 
As a 3rd-year Electronics and Communication Engineering student with basic knowledge of Digital Logic and FSM design, I realize that processing high-frequency telemetry streams using JavaScript on a CPU introduces software overhead, especially if the fleet size scales up or if we introduce complex vision data alongside telemetry.

### Hypothesis
If we can offload the telemetry parsing, filtering, and normalization pipeline to dedicated hardware (an FPGA), we can achieve deterministic, sub-microsecond latency and free up the main processor. 

### Expected Outcome
I intend to start by designing a basic hardware accelerator pipeline on an FPGA that can ingest telemetry packets, extract relevant fields, and output normalized data.

### Next Experiments
- Review literature on FPGA-based network processing and telemetry acceleration.
- Sketch out a basic architecture for a hardware telemetry packet parser.
- Start designing a basic FSM in Verilog to recognize packet headers.

### Future TODO
